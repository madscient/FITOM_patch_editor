# CLAUDE.md

このファイルは Claude Code がセッション開始時に自動的に読み込む、
このリポジトリ専用のコンテキストです。**このプロジェクトは複数の
マシンで並行して開発します。** 会話履歴やローカルディスク上の一時
状態はマシン間で共有されないため、「今どこまで進んでいて、次に何を
すべきか」は必ずこのリポジトリ内のドキュメント(このファイル +
`docs/STATUS.md` + `docs/DESIGN.md`)だけを見て判断できるようにして
あります。作業を始める前に、必ず下記「セッション開始時に読むもの」を
読んでください。

---

## 利用者とのやり取り

- 利用者への報告・説明・応答は**日本語**で行うこと。コード内コメントや
  コミットメッセージも日本語(このリポジトリの既存の慣習通り)。

## GUIの動作確認について

- GUI(`fitom_patch_editor_gui`)の実際のクリック操作(`SetCursorPos`+
  `PostMessage`によるウィンドウ自動操作、`docs/DESIGN.md` D-015参照)は
  **利用者から明示的に指示されない限り実施しない**。実マシンのマウス/
  フォーカスを奪うため、利用者が同じマシンで別の作業をしている場合に
  競合する。
- 代わりに、ビルド・`ctest`・(必要なら)キオスクモード起動
  (`fitom_patch_editor_gui.exe <profile> <hwbank-file> <prog>`、D-026)
  でのスクリーンショット1枚程度の受動的確認までに留め、それ以上の
  詳細な見た目確認(フォールドアウトを開く、値を変更するなど)は
  利用者自身の目視確認に委ねてよい。その場合は `docs/STATUS.md` に
  「クリック確認は未実施(利用者の目視確認待ち)」等、何が確認済みで
  何がそうでないかを正直に記録すること。

## プロジェクト概要

FITOM_X(別プロジェクト、音源エンジン本体)用のオフラインパッチ/
プロファイルエディタです。FITOM_Xが使うプロファイル/バンクJSON
(`*.profile.json` 等)を、FITOM_X自体を起動せずに直接読み書きします。
GUIはFITOM_X本体と同じくDear ImGui(GLFW + OpenGL3バックエンド)。
詳しい構成・設計判断・現在の実装状況は以下を参照してください。

- **`README.md`** — プロジェクト概要、ディレクトリ構成、閲覧・編集
  階層のデータモデルへのマッピング、ビルド方法。**利用者向け**の説明。
- **`docs/DESIGN.md`** — アーキテクチャと、なぜそう決めたかの判断ログ。
  同じ議論を繰り返さないための記録。
- **`docs/STATUS.md`** — 何が実装済みで何が未実装か、既知の課題、
  セッションごとの進捗ログ。

## セッション開始時に読むもの(この順番で)

1. `git pull`(またはそれに相当する最新化)をまず行う。
2. このファイル(`CLAUDE.md`)全体。
3. `docs/STATUS.md` の「進捗ログ」— 直近のセッションで何が行われ、
   次に何をする予定だったかを確認する。
4. 着手するタスクに関係する範囲で `docs/DESIGN.md` / `README.md`。

チャット履歴・メモリに頼らず、常にこの3点セットを一次情報源として
扱ってください。別マシンの別セッションでの作業内容は、コミットされて
いない限り一切引き継がれません。

## ビルド・テスト コマンド

```bash
# 依存関係(nlohmann-json, imgui, glfw3, glew)は vcpkg マニフェストモード
# (vcpkg.json)で解決する。VCPKG_ROOT 環境変数が必要(後述「マシンごとに
# 異なる設定」参照)。

cmake --preset vcpkg                 # Linux/macOS
cmake --preset vcpkg-windows         # Windows, Visual Studio 2022
cmake --preset vcpkg-windows-vs2026  # Windows, Visual Studio 2026 (CMake 4.2以上が必要)

cmake --build build/vcpkg            # プリセットの binaryDir に合わせて変える
ctest --test-dir build/vcpkg --output-on-failure   # fpe_data のスモークテスト(85項目)
```

GUIを含めずデータモデル層だけビルドしたい場合は `-DFPE_BUILD_GUI=OFF`。

## 複数マシン開発のためのルール

このプロジェクトは1人の開発者が複数のマシンで作業する想定です。
Claude Codeにはマシンをまたいだ記憶がないため、**gitリポジトリの
コミット済みの内容だけが、マシン間で共有される唯一の情報源**です。
以下を徹底してください。

### 1. 唯一の真実の情報源(Single Source of Truth)はgitリポジトリ

- 「何が完了していて、次に何をするか」の判断材料は、常に
  `docs/STATUS.md` の進捗ログと、実際のコミット履歴・コード自体から
  得ること。特定のマシンの会話履歴やローカルの一時ファイルだけに
  存在する情報を前提に作業しない。
- 逆に言えば、次のセッション(別マシンかもしれない)に引き継ぎたい
  情報は、**必ずコミットする**。「あとで別マシンでも同じ会話を続ける
  から大丈夫」という前提を置かない(続けられない)。

### 2. セッション終了時のルール

作業を中断・終了する際は、次を行ってから終える。

1. `docs/STATUS.md` の「進捗ログ」に、日付・行ったこと・次にやること
   を追記する(下記フォーマット参照)。未完了・動作未確認の部分は
   正直に「未検証」「壊れている」と明記する。
2. 重要な技術判断(ライブラリ選定、アーキテクチャ変更など)をした
   場合は `docs/DESIGN.md` に理由つきで記録する。
3. `git add` → `git commit` → `git push` まで行う。**作業が中途半端
   でもコミットして良い**(WIPコミットで構わない)。ローカルにしか
   存在しない変更を残したままセッションを終えない。

### 3. マシンごとに異なる設定はコミットしない

以下はマシン固有なので、リポジトリにハードコードしない・コミットしない。

- `VCPKG_ROOT` 環境変数(vcpkgのチェックアウト先)。各マシンで個別に
  設定する。`CMakePresets.json` は `$env{VCPKG_ROOT}` を参照するのみ。
- 利用可能なVisual Studioのバージョン(マシンによって2022/2026が
  違う可能性がある)。`vcpkg-windows` / `vcpkg-windows-vs2026` の
  どちらのプリセットが使えるかはマシン依存。
- `build/` ディレクトリ(`.gitignore` 済み)。

### 4. 技術判断は理由つきで書き残す

「なぜSDL2ではなくGLFWにしたか」「なぜgit submoduleではなくvcpkgに
したか」のような判断は、コード的には後から見えないため、
`docs/DESIGN.md` に判断ログとして残してあります。新しい判断をする
ときも同様に追記し、別マシンのセッション(≒このファイルしか読めない
まっさらな状態のClaude Code)が同じ議論を繰り返さないようにする。

### 5. 環境固有の落とし穴もすぐ記録する

このリポジトリの置き場所(ネットワーク越し/クラウド同期のドライブ等)
によっては、書き込み中のファイルが不完全な状態で保存されたり、
`git`のインデックスが壊れたりする事象が実際に発生しています(詳細は
`docs/DESIGN.md`の該当節)。ファイルを書き換えた後は
`wc -l`/`tail`/`git diff`等で内容が壊れていないか必ず確認する習慣を
つけてください。同様の問題に遭遇したら、その都度ここか
`docs/DESIGN.md`に追記してください。

### 進捗ログのフォーマット(`docs/STATUS.md`に追記する際)

```
### YYYY-MM-DD (マシン/作業者メモ、任意)
- やったこと: ...
- 未完了・既知の問題: ...
- 次にやること: ...
```

## コーディング規約(このプロジェクトで既に確立しているもの)

- C++17。`nlohmann::json` の `to_json`/`from_json` フリー関数パターン
  (ADL)を各構造体に用意する。
- JSON読み込みは緩く(欠落フィールドはドキュメント記載のデフォルト値
  にフォールバック)、書き込みは明示的に(正規のフィールド一式を常に
  出力)。詳細は `README.md` の「設計上のポイント」。
- サードパーティ依存はすべて vcpkg マニフェストモード
  (`vcpkg.json`)経由。リポジトリにベンダリング・submodule化しない
  (理由は `docs/DESIGN.md` 参照)。
- 各構造体・フィールドには、FITOM_X側のどのドキュメントを根拠にしたか
  コメントで注記する(未確認の推測箇所は `NOTE ON CONFIDENCE` と明記)。
